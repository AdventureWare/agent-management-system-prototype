# Challenge Submission Packager

## When to use this skill

- Use this skill when the user wants to enter a challenge, competition, or prize opportunity.
- Use this skill when a working prototype exists but the submission package is still vague.
- Use this skill when the real risk is missing criteria, weak positioning, or deadline chaos rather than building more features.

## Workflow

1. Extract the exact rules.
   Capture judging criteria, eligibility, required assets, deadlines, submission format, and any disqualifiers.
2. Choose the strongest angle.
   Decide what story gives the best fit with the criteria: technical novelty, customer value, social impact, market potential, execution quality, or speed.
3. Map required assets.
   Usually this includes some mix of:
   - written narrative
   - pitch or summary
   - demo script
   - screenshots or video
   - repo or technical explanation
   - team or project metadata
4. Build the backward plan.
   Start from the deadline and define checkpoints for final demo readiness, draft review, asset capture, and polish.
5. Pressure-test the proof.
   Check whether the demo and claims visibly support the chosen story.
6. Produce the final package outline.
   End with a submission checklist, missing items, and the next highest-value task.

## Output shape

- Rules and criteria summary
- Recommended angle
- Required asset list
- Backward timeline
- Draft outline for the written submission
- Demo plan
- Final checklist

## Failure shields

- Do not overbuild features that are invisible to judges.
- Do not make claims that the demo or artifacts cannot support.
- Treat submission format and deadline requirements as hard constraints.
- If eligibility or rules are unclear, resolve that before deeper work.
