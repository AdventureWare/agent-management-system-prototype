# MVP Scope Validation

## When to use this skill

- Use this skill when the user has an idea but not a disciplined scope.
- Use this skill when a feature is starting to sprawl and needs a smallest useful version.
- Use this skill when the team needs a brief that builders and reviewers can execute against.

## Workflow

1. Name the user and the pain.
   State who the MVP is for, what job they are trying to do, and what currently blocks them.
2. Define the one core loop.
   Reduce the concept to the primary workflow that must work end to end.
3. Cut aggressively.
   Separate:
   - required for first proof
   - useful soon after
   - explicitly out of scope
4. Define the proof standard.
   Write success criteria that are observable, not aspirational.
5. Add the measurement layer.
   Specify what needs to be tracked so the result can be evaluated after launch or testing.
6. Surface the main risks.
   Name the sharpest product, technical, and adoption risks.
7. End with a build brief.
   Include the MVP statement, acceptance criteria, non-goals, and now, next, later breakdown.

## Output shape

- User and problem
- Core workflow
- MVP scope
- Non-goals
- Success criteria
- Instrumentation or evidence plan
- Risks
- Now, next, later

## Failure shields

- If there are multiple audiences, force a primary audience first.
- If success cannot be measured, stop and define what proof would look like.
- Avoid roadmap padding. The MVP should answer a real question, not satisfy every imagined edge case.
- Prefer one solid vertical slice over several partial surfaces.
