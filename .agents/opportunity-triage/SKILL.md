# Opportunity Triage

## When to use this skill

- Use this skill when the user is deciding whether something is worth doing now.
- Use this skill when there are multiple candidate bets and focus needs to be protected.
- Use this skill when an opportunity sounds exciting but the real question is expected leverage, not novelty.

## Workflow

1. Define the decision.
   Capture the exact opportunity, the desired decision, the deadline, and the likely cost in time or money.
2. Frame the opportunity in one sentence.
   State who it helps, what upside exists, and why it matters now.
3. Score the opportunity on the few dimensions that matter most:
   - strategic fit with the main product or long-term direction
   - leverage or reuse of code, assets, distribution, or knowledge
   - speed to a meaningful proof point
   - expected upside such as revenue, prize money, distribution, or learning
   - execution difficulty and hidden dependency risk
   - evidence quality versus speculation
4. Identify the cheapest validating move.
   Prefer a small experiment, lightweight prototype, customer conversation, landing page, or rules read before committing to a full build.
5. Recommend one of:
   - pursue now
   - park with conditions
   - decline
6. End with a concrete next action.
   If the answer is not "pursue now," state what would need to become true to revisit it.

## Output shape

- Short decision summary
- Scorecard with reasoning
- Key risks or missing evidence
- Cheapest next test
- Final recommendation

## Failure shields

- Default to protecting focus on the main product unless the alternative has clearly superior leverage.
- Do not confuse personal interest with strategic fit.
- Do not recommend a large build when a smaller test can answer the real question.
- If the decision depends on missing external facts, say that directly and name the exact facts to collect.
